# 章节生成策略

本文档详细说明 knowledge-base 中每个章节的生成策略、使用工具和处理流程。

---

## 1. 身份与角色定义 (01-persona)

### 生成策略
该章节用于定义和记录系统/用户的身份、人格和偏好。内容应保持**权威性**和**稳定性**，不频繁变更。

### 使用工具
- **手动编辑**：直接使用 `write` 或 `edit` 工具更新 Markdown 文件
- **memory_search**：检索相关身份信息
- **会话启动流程**：每次会话自动加载 SOUL.md、IDENTITY.md、USER.md

### 处理流程
1. **初始化**：创建 SOUL.md（核心人格）、IDENTITY.md（身份标识）、USER.md（用户画像）
2. **更新**：当人格、身份或用户偏好发生变化时，手动编辑对应文件
3. **版本控制**：重大变更建议 commit 到 git，保留历史记录
4. **关联记忆**：重要变更应同步更新 MEMORY.md 的"用户偏好"部分

### 关键文件
- `SOUL.md` - 核心人格与价值观（语气、原则、边界）
- `IDENTITY.md` - 身份标识（名称、角色、emoji、头像）
- `USER.md` - 用户画像（姓名、 pronoun、时区、偏好）

### 注意事项
- 这些文件在每次会话启动时都会被读取，影响整体行为
- 修改后应立即告知用户，因为这是"灵魂"文件
- 保持简洁，避免过度定义

## 2. 项目文档 (02-projects)

### 生成策略
该章节用于存储**具体项目**的技术文档、架构设计、开发笔记等。内容应**具体、详细、可执行**。

### 使用工具
- **手动编辑**：创建/更新项目文档
- **coding-agent**：当项目需要代码生成时，该 agent 可自动创建项目文档
- **web_fetch**：当需要从网页抓取项目相关信息时使用
- **browser**：复杂网页交互时使用
- **github**：与 GitHub 仓库交互时使用

### 处理流程
1. **项目识别**：当用户提及新项目或 coding-agent 启动新任务时，在 `02-projects/` 下创建子目录
2. **文档结构**（推荐）：
   ```
   02-projects/项目名/
   ├── README.md            # 项目概览
   ├── ARCHITECTURE.md      # 架构设计
   ├── SETUP.md            # 环境配置
   ├── API.md              # API 文档（如适用）
   └── NOTES.md            # 开发笔记
   ```
3. **内容填充**：
   - coding-agent 完成任务后，自动生成/更新项目文档
   - 手动项目：用户或 assistant 直接创建文档
4. **关联任务**：在 `03-tasks/` 中为项目创建追踪任务
5. **定期回顾**：项目状态变更时（上线、暂停、废弃）更新 README.md

### 自动化机会
- coding-agent 结束后，自动提取关键信息生成 README.md
- 定期扫描项目目录，更新 MEMORY.md 中的项目状态

### 注意事项
- 区分 `02-projects/`（我的项目）和用户自己的项目目录
- 敏感信息（API keys、密码）不要写入，使用环境变量引用
- 技术文档应保持最新，陈旧文档应移到 `05-archive/`

## 3. 任务追踪 (03-tasks)

### 生成策略
该章节用于**追踪任务状态**（进行中/已完成）。内容应**简洁、可操作、有明确状态**。

### 使用工具
- **手动编辑**：创建/更新任务文件
- **coding-agent**：子代理完成任务后，自动创建任务完成记录
- **subagents**：管理子代理任务状态
- **sessions_send**：跨会话任务通知

### 处理流程
1. **任务创建**：
   - 用户在对话中提出新任务 → 在 `active/` 中创建 `.md` 文件
   - 文件名格式：`YYYY-MM-DD-任务简述.md` 或 `任务ID-简述.md`
   - 包含字段：任务描述、优先级、截止日期（如有）、关联项目

2. **任务分派**：
   - 简单任务：直接执行，完成后移到 `completed/`
   - 复杂任务：`sessions_spawn` 启动子代理，传入任务描述
   - 子代理执行过程中，定期更新任务状态（使用 `sessions_send` 通知主会话）

3. **任务更新**：
   - 子代理完成子任务时，使用 `memory_get` 读取任务文件，追加进度
   - 主会话定期检查 `active/` 目录，主动询问任务进展

4. **任务完成**：
   - 所有子任务完成 → 在任务文件中标记 ✅ 完成
   - 将文件移到 `completed/`，保留历史
   - 在 MEMORY.md 中记录完成经验（如适用）

5. **任务取消/延期**：
   - 移动到 `archive/` 或添加 `#cancelled` 标签
   - 记录取消原因

### 任务文件模板
```markdown
# 任务：XXX

**状态**: #active / #completed / #cancelled
**创建时间**: 2026-03-10
**优先级**: high / medium / low
**关联项目**: 02-projects/英语学习平台

## 描述
任务的具体描述，包括背景、目标、验收标准。

## 子任务
- [ ] 子任务1
- [ ] 子任务2

## 进度
- 2026-03-10 20:00: 开始执行，已完成子任务1
- 2026-03-10 21:00: 子任务2进行中...

## 备注
任何补充信息
```

### 自动化机会
- 启动子代理时自动创建任务文件
- 子代理定期向 `03-tasks/active/` 写入进度
- 会话启动时自动提醒所有 `active` 任务

### 注意事项
- `active/` 和 `completed/` 分开，便于快速查看进行中任务
- 任务文件应包含时间戳，便于追溯
- 复杂任务的子任务应在同一文件中管理，避免碎片化

## 4. 学习与经验 (04-learnings)

### 生成策略
该章节用于**记录学习笔记、技术经验、最佳实践**。内容应**精炼、可复用、有洞察**。

### 使用工具
- **手动编辑**：创建学习笔记
- **memory_search**：查找相关知识
- **coding-agent**：编码过程中发现的技巧，自动生成学习笔记
- **self-improving**：自我反思后，提取经验教训

### 处理流程
1. **学习触发**：
   - 解决问题后，主动总结（"这次遇到了 X 问题，原因是 Y，解决方法是 Z"）
   - 阅读文档/代码后，提炼关键点
   - coding-agent 执行中发现的脚本技巧、工具用法

2. **笔记结构**：
   ```
   04-learnings/
   ├── 技术笔记/
   │   ├── Supabase-权限管理.md
   │   └── Playwright-选择框交互.md
   └── 最佳实践/
       ├── 子代理任务管理.md
       └── OpenRouter-限流处理.md
   ```

3. **内容规范**：
   - **问题**：遇到的具体问题
   - **原因**：根本原因分析
   - **解决方案**：具体步骤、代码片段
   - **经验**：可复用的模式/反模式
   - **参考**：相关文档链接、issue 链接

4. **与 MEMORY.md 同步**：
   - 重要的经验教训应提炼后写入 MEMORY.md 的"经验"部分
   - 避免简单重复，MEMORY.md 只保留**最核心**的经验

### 标签系统
- `#bug` - 解决过的 bug
- `#tip` - 实用技巧
- `#pattern` - 可复用模式
- `#anti-pattern` - 应避免的做法
- `#ref` - 参考资料

### 自动化机会
- coding-agent 结束后，自动分析日志，提取 error 和解决方式
- 定期（每周）回顾 `memory/` 日志，将有价值的信息迁移到 `04-learnings/`

### 注意事项
- 学习笔记应**重事实、轻叙述**，便于未来快速检索
- 每个笔记应独立，避免一个文件包含多个不相关主题
- 过时的经验应标记 `#deprecated` 或移到 `05-archive/`

## 5. 归档资料 (05-archive)

### 生成策略
该章节用于**存放不再活跃但需要保留的历史资料**。内容应**完整但不可变**。

### 使用工具
- **手动移动**：将旧资料移到 archive
- **自动归档**：定期任务（cron）自动归档 `active/` 中超过 90 天的已完成任务

### 处理流程
1. **归档触发**：
   - 项目终止/暂停 → 整个项目目录移到 `05-archive/projects/`
   - 任务完成超过 90 天 → 自动移到 `05-archive/tasks/`
   - 过时的学习笔记 → 移到 `05-archive/learnings/` 并添加 `#deprecated`

2. **归档结构**：
   ```
   05-archive/
   ├── projects/          # 已归档项目
   │   └── 旧项目名/
   ├── tasks/             # 已归档任务
   │   ├── 2024-Q1/
   │   └── 2024-Q2/
   └── learnings/         # 已归档学习笔记
   ```

3. **保留策略**：
   - 永远保留，但不活跃搜索
   - `memory_search` 默认排除 `05-archive/`（需显式指定）
   - 物理存储不变，只是逻辑上"冷数据"

### 注意事项
- 归档是**单向**的，不要轻易从 archive 恢复到 active
- 归档前应确保关键信息已提炼到 MEMORY.md 或 `04-learnings/`
- 定期（每年）清理无价值的历史资料（可物理删除）

## 通用流程：知识入库

无论是哪个章节，新知识的入库都应遵循以下流程：

### 1. 识别
- 对话中用户提到"记住这个"
- 或 assistant 自我反思：这个经验值得保留

### 2. 分类
判断知识类型：
- 身份/人格 → `01-persona/`
- 项目文档 → `02-projects/`
- 任务/待办 → `03-tasks/`
- 经验/技巧 → `04-learnings/`
- 其他 → 判断是否归档

### 3. 格式化
- 使用 Markdown
- 包含时间戳和标签
- 简明扼要，避免冗长

### 4. 写入
使用 `write` 或 `edit` 工具，路径相对于 workspace 根目录：
```bash
write("/root/.openclaw/workspace/knowledge-base/04-learnings/技术笔记/新技能.md", content)
```

### 5. 索引（可选）
如果知识非常重要，更新 `knowledge-base/README.md` 或 MEMORY.md 使其可见。

### 6. 通知
如果用子代理生成知识，完成后通过 `sessions_send` 通知主会话：
> "已创建学习笔记：04-learnings/技术笔记/XXX.md"

## 工具矩阵

| 章节 | 主要工具 | 辅助工具 | 自动化建议 |
|------|---------|---------|-----------|
| 01-persona | write/edit | memory_search | 会话启动自动加载 |
| 02-projects | write/edit | coding-agent, github | agent 结束后自动生成文档 |
| 03-tasks | write/edit | subagents, sessions_send | 任务状态 push 通知 |
| 04-learnings | write/edit | self-improving, memory_search | 定期从 memory/ 迁移精华 |
| 05-archive | mv (手动) | - | cron 自动归档超时任务 |

## 最佳实践

1. **原子性**：一个文件一个主题，不要混杂
2. **可追溯**：包含时间戳和作者（如适用）
3. **标签化**：使用 `#tag` 标记状态和类型
4. **链接化**：用相对链接关联相关文档
5. **定期回顾**：
   - 每月检查 `03-tasks/active/`，清理过期任务
   - 每季度回顾 `04-learnings/`，提炼精华到 MEMORY.md
   - 每年归档 `05-archive/`
