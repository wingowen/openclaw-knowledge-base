## 二、交互方式：HTTP REST API

两个系统通过 **HTTP REST API** 通信，主系统是调用方（Client），爬虫服务是被调用方（Server）。

### 连接配置

| 配置项 | 主系统 | 爬虫服务 |
|--------|--------|----------|
| 环境变量 | `clawer.api.url` | 无（被动接受） |
| 默认地址 | `http://127.0.0.1:5000` | `0.0.0.0:5000` |
| 通信协议 | HTTP POST/GET | HTTP POST/GET |
| 数据格式 | JSON | JSON |

## 三、核心交互接口详解

### 3.1 任务提交接口（最核心）

**接口**: `POST /api/local/tasks`
**调用方**: `common/tasks.py` 中的多个 Celery 任务
**行为**: 同步执行爬虫脚本，等待完成后返回结果

```python
# 主系统调用示例（来自 common/tasks.py）
import requests

response = requests.post(
    f"{settings.CLAWER_API_URL}/api/local/tasks",
    json={
        "script_name": "step_one_v2.py",    # 爬虫脚本名
        "params": {                          # 传给脚本的参数
            "industry_id": 123,
            "keyword": "人工智能",
            "page_num": 10
        }
    },
    timeout=3600  # 1小时超时
)

result = response.json()
# { "success": true, "output": "...", "return_code": 0 }
```

**爬虫服务处理流程**:
1. 验证脚本名（防路径遍历攻击：拒绝包含 `..` 或 `/` 的路径）
2. 在 `scripts/` 目录下查找脚本
3. `subprocess.Popen([python, script_path, json_params])` 直接执行
4. 实时读取 stdout
5. 返回执行结果（JSON）

### 3.2 异步任务接口

**接口**: `POST /api/tasks`
**行为**: 提交到爬虫服务的 Celery 队列异步执行

```python
response = requests.post(
    f"{settings.CLAWER_API_URL}/api/tasks",
    json={
        "script_name": "step_one_v2.py",
        "params": {...}
    }
)
# 返回: { "task_id": "abc-123", "status": "submitted" }

# 后续轮询状态
status = requests.get(f"{settings.CLAWER_API_URL}/api/tasks/{task_id}")
```

### 3.3 任务镜像同步

**接口**: `POST /api/task_mirror/upsert`
**用途**: 主系统将任务元数据同步到爬虫服务，便于双向追踪

```python
# 主系统 → 爬虫服务，同步任务状态
requests.post(
    f"{settings.CLAWER_API_URL}/api/task_mirror/upsert",
    json={
        "task_id": "celery-task-id",
        "task_type": "industry",      # industry | batch | task | industry_query
        "industry_id": 123,
        "status": "running",
        "metadata": {...}
    }
)
```

### 3.4 数据同步接口

**接口**: `GET /api/sync_data`
**用途**: 将爬虫数据库 (`test_asc`) 的数据同步到主系统数据库 (`test_as`)

```
爬虫 DB (test_asc)          主系统 DB (test_as)
┌─────────────────┐         ┌─────────────────┐
│ patent_data     │ ─sync─▶ │ patent_data     │
│ company_raw     │ ─sync─▶ │ company_longlist│
│ news_raw        │ ─sync─▶ │ news_classified │
└─────────────────┘         └─────────────────┘
```

### 3.5 产业查询插入

**接口**: `POST /api/insert_industry_query`
**用途**: 主系统将生成的搜索查询推送给爬虫服务

## 七、关键技术细节

### 7.1 为什么用 `/api/local/tasks` 而不是 `/api/tasks`？

主系统 **主要使用同步模式** (`/api/local/tasks`)，而非异步 Celery 模式：

- **原因**: 主系统自己已有 Celery 编排（`common/tasks.py`），在自己的 Celery 任务中调用爬虫服务时，需要等待爬虫完成后再继续后续处理（如数据清洗、评分）
- **效果**: 主系统 Celery Task → HTTP 同步调用 → 爬虫脚本执行 → 返回 → 主系统继续处理
- **优势**: 避免了两层异步嵌套（主系统 Celery + 爬虫 Celery），简化错误处理

### 7.2 路径安全

爬虫服务对脚本名做了安全校验：

```python
# api.py 中的安全检查
if '..' in script_name or '/' in script_name:
    return {"success": False, "error": "非法脚本路径"}, 403
```

### 7.3 超时设置

| 层级 | 超时 | 说明 |
|------|------|------|
| 主系统 Celery Task | 3 小时 (硬) / 2h50m (软) | 整个任务生命周期 |
| HTTP 请求 | 1 小时 | 单次爬虫调用 |
| 爬虫 Celery Task | 1 小时 | 爬虫服务内部 |

### 7.4 并发控制

- 主系统 `monitor_company_data_crawl` 使用 `ThreadPoolExecutor` 并发调用爬虫服务
- 天眼查 API 通过 Redis 控制账号并发（`account_redis.py`）
- 爬虫服务 Celery Worker 默认 1 并发

## 八、设计评价

### 优点

1. **职责分离清晰**: 主系统负责业务逻辑和 AI，爬虫服务专注数据采集
2. **接口简洁**: 统一的 `/api/local/tasks` 入口，脚本即插即用
3. **同步为主**: 避免双层异步的复杂性，错误处理更直观
4. **双库隔离**: 爬虫原始数据与业务数据分开，互不污染
5. **数据可追溯**: task_mirror 机制实现双向状态追踪

### 潜在风险

1. **单点依赖**: 主系统强依赖爬虫服务的可用性，爬虫服务宕机会阻塞整个流程
2. **长时间 HTTP 连接**: 同步模式下，一个爬虫任务可能持续数十分钟，连接稳定性是挑战
3. **数据一致性**: 双库同步存在延迟，极端情况下可能出现数据不一致
4. **无重试机制**: HTTP 调用失败后无自动重试（`MAX_RETRIES = 0`）

### 改进建议

1. 增加 HTTP 调用的重试和熔断机制
2. 考虑关键长时任务改为异步 + 回调模式
3. 数据同步增加事务保证和冲突检测
4. 爬虫服务增加健康检查和自动恢复
