## 一、核心发现

### 当前数据库配置

```
.env: db.name = 'test_as'   ← 主系统直连主库
```

**关键问题**：`sql_util.py` 的 `get_db_connection()` 只连接一个数据库（`test_as`），所有通过它执行的 SQL 都落在主库上。这意味着：
- 爬虫侧的表（`task`、`patent_task`、`patent_query`、`patent_patenter`、`industry_query`）在 `test_as` 中也有副本
- 数据流转依赖 `sync_client.py` 的 HTTP 同步机制，但只有 `crawl_patent_task` 调用了它

### 数据流现状

```
理想状态（双库分离）:
  爬虫脚本 → test_asc → sync_client → test_as → 评分/报告

实际状态（大部分任务）:
  爬虫脚本 → test_as（直连）→ 评分/报告
  （跳过了 test_asc + 同步步骤）
```

## 四、影响评估

### 数据一致性风险

由于大部分任务直接读写 `test_as` 中的爬虫表副本，存在以下风险：

1. **数据来源不明确**：`test_as` 中的 `task`、`patent_task` 等表数据可能来自：
   - 旧的同步机制
   - 直接写入（绕过爬虫服务）
   - 手动数据迁移

2. **双写冲突**：如果爬虫服务也在写 `test_as`（而非 `test_asc`），则双库架构形同虚设

3. **同步缺失**：只有专利爬取有同步流程，天眼查数据、新闻分类等都没有

### 重构优先级

| 优先级 | 任务 | 原因 |
|--------|------|------|
| P0 | `process_patent_data` | 核心数据处理管道，直接影响后续所有任务 |
| P0 | `integrated_patenter_task` | 专利权人→企业转换，影响企业数据完整性 |
| P1 | `monitor_company_data_crawl` | 企业数据爬取，涉及 11 个维度 |
| P1 | `extract_patent_tags` | 专利标签提取，影响报告质量 |
| P2 | `generate_industry_report` | 读取的数据已在主库，风险较低 |
| P2 | `generate_company_reports` | 读取的数据已在主库，风险较低 |
| P3 | `check_all_downstream_tasks` | 状态检查，不涉及核心数据 |
| P3 | `orchestrate_company_report_flow` | 纯编排逻辑 |

## 五、重构建议

### 短期方案（最小改动）

为每个需要读取爬虫数据的任务添加同步步骤：

```python
# 在 process_patent_data 开头添加
from data_processing.sync_client import sync_table_data

# 同步必要的爬虫表
sync_table_data('patent_task')
sync_table_data('patent')
```

### 长期方案（架构改造）

1. **引入双数据库连接管理器**：
   ```python
   # sql_util.py
   def get_crawler_db_connection():
       """连接爬虫库 test_asc"""
       return pymysql.connect(database='test_asc', ...)
   
   def get_main_db_connection():
       """连接主库 test_as"""
       return pymysql.connect(database='test_as', ...)
   ```

2. **按数据来源分离读写**：
   - 爬虫产生的数据 → 写入 `test_asc` → 同步到 `test_as`
   - 主系统业务数据 → 直接读写 `test_as`

3. **统一同步入口**：创建一个 `DualDBManager` 类，封装所有跨库操作
